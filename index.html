发表于 2021-04-21 分类于 Linux
本文字数： 987 阅读时长 ≈ 1 分钟

大概过程

当用户进程执行完库函数 recvfrom（Glibc库）调用以后，用户进程就通过系统调用 sys_recvfrom 陷入到内核态工作，如果接收队列没有数据，进程就会进程睡眠态被系统挂起，等待资源到来。

内核收包前的准备

  1. 创建 ksoftirqd 线程，并设置好对应的线程函数，用来处理软中断
  2. 协议栈注册，linux会实现许多协议，比如arp，icmp，ip，udp，tcp，每一个协议都会将自己的处理函数向内核进行注册一下，方便数据包到来后迅速找到对应的处理函数
  3. 网卡驱动初始化，每个驱动都有一个初始化函数，在这个初始化过程中，网卡驱动会初始化DMA，注册ethtool实现函数，初始化NAPI，注册poll函数等
  4. 启动网卡，分配RX,TX队列，注册中断对应的处理函数

ready，打开硬中断，等待数据包的到来

数据包到来后的处理过程

  1. 网卡将数据帧通过DMA的方式copy到内存的RingBuffer中，然后向CPU发起（硬）中断通知
  2. CPU响应中断请求，调用网卡启动时注册的中断处理函数
  3. 硬中断处理过程非常短，只是记录了一个寄存器，修改了一下CPU的poll_list，然后发出一个软中断
  4. 内核线程ksoftirqd线程发现有软中断请求到来，先关闭硬中断
  5. ksoftirqd线程开始调用驱动的poll函数收包
  6. poll函数将收到的包送到协议栈注册的ip_rcv函数中（IP协议层处理）
  7. ip_rcv函数再讲包送到udp_rcv函数中（对于tcp包就送到tcp_rcv）（UDP协议层处理）
  8. 处理完以后，判断该socket是否正在被占用，如果没有，那就可以直接放到socket的接收队列中，如果有，那就通过sk_add_backlog把数据包添加到backlog队列
  9. 资源到位，用户进程被唤醒

差分隐私算法

差分隐私（Differential Privacy）：保护的是数据源中一点微小改动导致的隐私泄露问题，假设给定两个数据集D和D’，它们有且仅有一条数据不一样，这一条数据就属于差分隐私，DP算法就是保护这一条数据的隐私不被暴露出来。

数据加噪

对原始数据集进行加噪，敏感度为1，原始数据集为 [452, 477, 8, 1, 4, 6037]

  • 敏感度：指增/减一条数据对具体操作（如count/sum/avg等）造成的影响。该场景下各个区间段是互不相交的count结果，因此增/减一条数据影响某区间的统计数为1，即敏感度为1。
  • 加噪：向原始数据集中引入随机性，以满足差分隐私的思想，对于数值型的查询结果，常用的方法是向查询结果中加入服从拉普拉斯分布的噪声。
  • 拉普拉斯机制：

假设对原始数据加噪后的数据集为：

[452, 477, 8, 1, 4, 6037] -> [449.87, 475.66, 8.20, 1.11, 4.03, 6036.97]

可用性评估

对加噪处理后的数据进行可用性评估，根据设计的评估规则，若无法满足该评估规则则需要修改机制或调整参数重新进行加噪处理。

  • 为什么要可用性评估：

如果我设计一个算法，不管你输入是什么，我总是输出数字0，那这个算法也是满足DP的，因为它的输出跟输入压根没关系，当然不会泄露隐私。

比如原始数据的18-23年龄维度的占比是6.4%，而经过加噪处理后的占比到达了10%，超过了评估规定的阈值，那么该DP算法是不具有可用性，或者可用性非常差

取整处理

如果用的是浮点型的加噪算法，需要对加噪后的小数点进行处理，假设采用下面的处理逻辑：

  • 若加噪后比对应原始数据大，直接舍弃小数位，否则小数位直接进位

则取整后的数据集为：

[449.87, 475.66, 8.20, 1.11, 4.03, 6036.97] -> [450, 476, 8, 1, 4, 6037]

一致性处理

对加噪后的数据集进行规整，保证加噪前后各标签内的所有属性总和一致

  • 原始数据：[452, 477, 8, 1, 4, 6037] -> 总和：6979
  • 加噪后数据：[450, 476, 8, 1, 4, 6037] -> 总和：6976

计算两个数据集的差值 a = 6979-6976 = 3，从第一项开始加/减 1（a > 0 时 -1，a < 0时 + 1），同时 a - 1，重复执行直到 a=0 为止。

则规整后的数据集为：

[450, 476, 8, 1, 4, 6037] -> [449, 475, 7, 1, 4, 6037]

返回差分隐私后的结果集

Return [449, 475, 7, 1, 4, 6037]

对比下画像维度下（18-23）的占比：

  • 原数据集：452/(8+452+477+6037+4+1)= 452/6979 = 6.48 %
  • DP后的数据集：449/(8+452+477+6037+4+1)= 449/6979 = 6.43 %

内核空间和用户空间

内核空间和用户空间是现代操作系统的两种工作模式，内核模块运行在内核空间，而用户态应用程序运行在用户空间。它们代表不同的级别，而对系统资源具有不同的访问权限。内核模块运行在最高级别（内核态），这个级下所有的操作都受系统信任，而应用程序运行在较低级别（用户态）。在这个级别，处理器控制着对硬件的直接访问以及对内存的非授权访问。内核态和用户态有自己的内存映射，即自己的地址空间。

处理器总处于以下状态中的一种：
  • 内核态，运行于进程上下文，内核代表进程运行于内核空间；
  • 内核态，运行于中断上下文，内核代表硬件运行于内核空间；
  • 用户态，运行于用户空间。

用户空间的应用程序，通过系统调用，进入内核空间。由内核代表该进程运行于内核空间，这就涉及到上下文的切换，用户空间和内核空间具有不同的地址映射，通用或专用的寄存器组，而用户空间的进程要传递很多变量、参数给内核，内核也要保存用户进程的一些寄存器、变量等，以便系统调用结束后回到用户空间继续执行。

所谓的“进程上下文”，就是一个进程在执行的时候，CPU的所有寄存器中的值、进程的状态以及堆栈上的内容，当内核需要切换到另一个进程时，它需要保存当前进程的所有状态，即保存当前进程的进程上下文，以便再次执行该进程时，能够恢复切换时的状态，继续执行。

硬件通过触发信号，导致内核调用中断处理程序，进入内核空间。这个过程中，硬件的一些变量和参数也要传递给内核，内核通过这些参数进行中断处理。

所谓的“中断上下文”，其实也可以看作就是硬件传递过来的这些参数和内核需要保存的一些其他环境（主要是当前被中断的进程环境）。

上下文context：上下文简单说来就是一个环境，相对于进程而言，就是进程执行时的环境。具体来说就是各个变量和数据，包括所有的寄存器变量、进程打开的文件、内存信息等。

一个进程的上下文可以分为三个部分:用户级上下文、寄存器上下文以及系统级上下文。

  • 用户级上下文: 正文、数据、用户堆栈以及共享存储区；
  • 寄存器上下文: 通用寄存器、程序寄存器(IP)、处理器状态寄存器(EFLAGS)、栈指针(ESP)；
  • 系统级上下文: 进程控制块task_struct、内存管理信息(mm_struct、vm_area_struct、pgd、pte)、内核栈

当发生进程调度时，进行进程切换就是上下文切换(context switch)。操作系统必须对上面提到的全部信息进行切换，新调度的进程才能运行。而系统调用进行的是模式切换(mode switch)。模式切换与进程切换比较起来，容易很多，而且节省时间，因为模式切换最主要的任务只是切换进程寄存器上下文的切换。

进程上下文主要是异常处理程序和内核线程。内核之所以进入进程上下文是因为进程自身的一些工作需要在内核中做。例如，系统调用是为当前进程服务的，异常通常是处理进程导致的错误状态等。

内核进入中断上下文是因为中断信号而导致的中断处理或软中断。而中断信号的发生是随机的，中断处理程序及软中断并不能事先预测发生中断时当前运行的是哪个进程，事实上，对于A进程希望等待的中断信号，可能在B进程执行期间发生。例如，A进程启动写磁盘操作，A进程睡眠后B进程在运行，当磁盘写完后磁盘中断信号打断的是B进程，在中断处理时会唤醒A进程。

内核可以处于两种上下文：进程上下文和中断上下文。

  • 在系统调用之后，用户应用程序进入内核空间，此后内核空间针对用户空间相应进程的代表就运行于进程上下文。
  • 异步发生的中断会引发中断处理程序被调用，中断处理程序就运行于中断上下文。
  • 中断上下文和进程上下文不可能同时发生。

运行于进程上下文的内核代码是可抢占的，但中断上下文则会一直运行至结束，不会被抢占。因此，内核会限制中断上下文的工作，不允许其执行如下操作：

  1. 进入睡眠状态或主动放弃CPU;

由于中断上下文不属于任何进程，它与current没有任何关系（尽管此时current指向被中断的进程），所以中断上下文一旦睡眠或者放弃CPU，将无法被唤醒。所以也叫原子上下文（atomic context）。

  1. 占用互斥体;

为了保护中断句柄临界区资源，不能使用mutexes。如果获得不到信号量，代码就会睡眠，会产生和上面相同的情况，如果必须使用锁，则使用spinlock。

  1. 执行耗时的任务;

中断处理应该尽可能快，因为内核要响应大量服务和请求，中断上下文占用CPU时间太长会严重影响系统功能。在中断处理例程中执行耗时任务时，应该交由中断处理例程底半部来处理。

  1. 访问用户空间虚拟内存。

因为中断上下文是和特定进程无关的，它是内核代表硬件运行在内核空间，所以在中断上下文无法访问用户空间的虚拟地址

  1. 中断处理例程不应该设置成reentrant（可被并行或递归调用的例程）。

因为中断发生时，preempt和irq都被disable，直到中断返回。所以中断上下文和进程上下文不一样，中断处理例程的不同实例，是不允许在SMP上并发运行的。

  1. 中断处理例程可以被更高级别的IRQ中断。

如果想禁止这种中断，可以将中断处理例程定义成快速处理例程，相当于告诉CPU，该例程运行时，禁止本地CPU上所有中断请求。这直接导致的结果是，由于其他中断被延迟响应，系统性能下降。

Linux内核介绍

通常一个内核的组成：

  • 负责响应中断的中断服务程序
  • 负责管理多个进程从而分享处理器时间的调度程序
  • 负责管理进程地址空间的内存管理程序
  • 网络
  • 进程间通信
  • …

每个处理器在任何指定时间点上的活动必然概括为下列三者之一：

  • 运行于用户空间，执行用户进程
  • 运行于内核空间，处于进程上下文，代表某个特定的进程执行
  • 运行内核空间，处于中断上下文，与任何进程无关，处理某个特定的中断

进程管理

进程

进程就是处于执行期的程序，包括：打开的文件，挂起的信号，内核内部数据，处理器状态，一个或多个具有内存映射的内存地址空间及一个或多个执行线程，用来存放全局变量的数据段等。

每个进程都拥有一个独立的程序计数器、进程栈和一组进程寄存器。

  • 任务队列：存放进程列表的双向循环列表
  • 进程描述符：链表中的每一项，类型为task_struct，包含一个具体进程的所有信息

进程创建

  • fork()通过拷贝当前进程创建一个子进程
  • exec()函数负责读取可执行文件并将其载入到地址空间开始运行

线程在Linux中的实现

Linux把所有的线程都当做进程来实现；线程的创建和普通进程的创建类似，只不过在调用clone()的时候需要传递一些参数标志来指明需要共享的资源；

进程终结

一般来说，进程的析构发生在进程调用exit()系统调用时；调用exit_notify()向父进程发送信号，给子进程重新找养父，养父为线程组中的其他线程或者为init进程；

进程调度

linux采用两种不同的优先级范围，划分两种进程：

  • 普通进程   -20 ~ +19   nice-小   优先级-大
  • 实时进程   0 ~ 99   nice-大   优先级-大

调度器以模块方式提供，允许不同类型的进程可以有针对地选择调度算法：

  • 普通进程-SCHED_NORMAL（完全公平调度-红黑树）
  • 实时进程-SCHED_FIFO 和 SCHED_RR

系统调用

在Linux中，系统调用是用户空间访问内核的唯一手段；除异常和陷入外，它们是内核唯一的合法入口。

这些系统调用提供的接口让应用程序受限地访问硬件设备，提供了创建新进程并与已有进程进行通信的机制，也提供了申请操作系统其他资源的能力；

linux执行系统调用的连锁反应：陷入内核，传递系统调用号和参数，执行正确的系统调用函数，并把返回值带回用户空间；

内核数据结构

链表，队列，映射，二叉树

中断和中断处理

中断本质上是一种特殊的电信号，有硬件设备发出，通过中断控制器发向处理器，在响应一个特定中断时，内核会执行一个函数，中断处理程序；

进程上下文和中断上下文：

  • 进程上下文：可以通过current宏关联当前进程，可以睡眠，也可以调用调度程序；
  • 中断上下文：具有较为严格的时间限制，打断其他代码；

下半部和推后执行的工作

  • 上半部：对接收的中断进行应答或者复位硬件；
  • 下半部：处理和操作数据包的其他工作；
    • 软中断：中断；
    • tasklets：中断，同类型不能同时执行；
    • 工作列队；内核进程，在进程上下文中执行；

内核同步

临界区，死锁，锁的争用，加锁粒度；

  • 原子操作：一组针对整数（atomic_t类型-在int类型的低8位嵌入了一个锁）进行操作，另一组对单独的位进行操作；
  • 自旋锁；最多只能被一个可执行线程持有，忙循环-旋转等待锁重新可用，不可递归，读-写自旋锁；
  • 信号量：如果有一个任务试图获得一个不可用的信号量时，信号量会将其推进一个等待队列，然后让其睡眠，信号量比自旋锁有更大的开销；

定时器和时间管理

相对于事件驱动，内核中大量的函数都是基于时间驱动的；全局变量jiffies用来记录自系统启动以来产生的节拍的总数；

体系结构提供了两种设备进行计时：

  • 实时时钟RTC：用来持久存放系统时间的设备，在系统启动时初始化xtime变量；
  • 系统定时器：内核在启动时对PIT（可编程中断时钟）进行编程初始化，使其以HZ/秒的频率产生时钟中断；

定时器（有时也称动态定时器或者内核定时器）：管理内核流逝的时间的基础–延迟执行；指定的函数将在定时器到期自动执行，在超时后自行撤销，运行次数不受限制；

内存管理

  • 页分配器：分配器标志（行为修饰器、区修饰器、类型）；页、区；硬件设备用到的任何内存区都必须是物理上连续的块，软件使用的内存块是虚拟地址连续的内存块；
  • slab分配器：每种对象类型对应一个高速缓存；每个高速缓存可以由多个slab组成；每个slab由一个或多个物理上连续的页组成；

虚拟文件系统

虚拟文件系统（VFS）作为内核子系统，为用户空间程序提供了文件和文件系统相关的接口；

VFS中有四个主要的对象类型：

  • 超级块对象：代表一个具体的已安装文件系统；
  • 索引节点对象：代表一个具体文件；
  • 目录项对象：代表一个目录项，是路径的一个组成部分；
  • 文件对象：代表由进程打开的文件（内存中的表示）；

块I/O层

块设备，字符设备；

  • 缓冲区：磁盘块在内存中的表示；
  • 缓冲区头：缓冲区对应的描述符，说明缓冲区到块的映射关系；
  • bio结构体：代表正在现场的以片段链表形式组织的块I/O操作；

请求队列—-先执行 合并和排序 的与操作；

I/O调度程序：

  • Linus电梯–最终期限I/O调度–预测I/O调度；
  • 完全公平的排队I/O调度（CFQ）；
  • 空操作I/O调度；

进程地址空间

内存区域可以包含各种内存对象，比如：

  • 代码段：可执行文件代码的内存映射；
  • 数据段：可执行文件的已初始化全局变量的内存映射；
  • 全局变量：包含未初始化全局变量，bss段的零页的内存映射；
  • 栈：用于进程用户空间栈；
  • 共享库、内存映射文件；

页高速缓存和页回写

  • 页高速缓存：①读操作过程：到底该缓存谁取决于谁被访问到；②写操作过程：不缓存；写透缓存；回写；
  • 缓存回收：①LRU回收策略；②双链策略；

设备与模块：块设备，字符设备，网络设备；
